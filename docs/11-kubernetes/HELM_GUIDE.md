## ğŸ“¦ Helm Concepts

### 1. Chart

A **Chart** is a collection of files that describe a Kubernetes application.

**Structure:**
```
electronic-paradise/
â”œâ”€â”€ Chart.yaml          # Chart metadata
â”œâ”€â”€ values.yaml         # Default values
â”œâ”€â”€ values-staging.yaml # Staging overrides
â”œâ”€â”€ values-prod.yaml    # Production overrides
â””â”€â”€ templates/          # Kubernetes templates
    â”œâ”€â”€ deployment.yaml
    â”œâ”€â”€ service.yaml
    â””â”€â”€ ...
```

### 2. Template

A **Template** is a YAML file with placeholders that get filled with values.

**Example:**
```yaml
# Template
spec:
  replicas: {{ .Values.services.auth.replicas }}
  containers:
    - image: {{ .Values.global.imageRegistry }}/{{ .Values.services.auth.image.repository }}:{{ .Values.services.auth.image.tag }}

# Generated YAML (with values)
spec:
  replicas: 2
  containers:
    - image: ghcr.io/rahulsharma2309/electronic-paradise-auth:latest
```

### 3. Values

**Values** are configuration data that customize the chart.

**values.yaml:**
```yaml
services:
  auth:
    replicas: 2
    image:
      tag: latest
```

**values-prod.yaml:**
```yaml
services:
  auth:
    replicas: 5  # More replicas for production
    image:
      tag: v1.0.0  # Specific version
```

### 4. Release

A **Release** is a running instance of a chart.

```bash
# Install creates a release
helm install electronic-paradise . -n staging
# Release name: electronic-paradise
# Chart: current directory (.)
# Namespace: staging
```

## ğŸš€ How to Use Our Chart

### Prerequisites

1. **Kubernetes cluster** (Docker Desktop, K3s, cloud)
2. **kubectl** configured
3. **Helm 3.x** (optional, but recommended)

### Installation

#### Staging Environment

```bash
# Navigate to chart directory
cd infra/k8s/helm/electronic-paradise

# Install to staging
helm install electronic-paradise . \
  --namespace staging \
  --create-namespace \
  -f values-staging.yaml

# Verify
helm list -n staging
kubectl get pods -n staging
```

#### Production Environment

```bash
# Install to production
helm install electronic-paradise . \
  --namespace prod \
  --create-namespace \
  -f values-prod.yaml
```

### Upgrading

```bash
# Upgrade staging
helm upgrade electronic-paradise . \
  --namespace staging \
  -f values-staging.yaml

# Upgrade production
helm upgrade electronic-paradise . \
  --namespace prod \
  -f values-prod.yaml
```

### Uninstalling

```bash
# Uninstall from staging
helm uninstall electronic-paradise --namespace staging

# Uninstall from production
helm uninstall electronic-paradise --namespace prod
```

## ğŸ“ Templates and Values

### Template Syntax

Helm uses Go templates with special syntax:

```yaml
# Variables
{{ .Values.services.auth.replicas }}

# Conditionals
{{- if .Values.services.auth.enabled }}
# ...
{{- end }}

# Loops
{{- range .Values.services }}
# ...
{{- end }}

# Functions
{{ include "helper.name" . }}
```

### Helper Functions

Our chart includes helper functions in `_helpers.tpl`:

```yaml
# Generate labels
{{- include "electronic-paradise.labels" . }}

# Generate image reference
{{- include "electronic-paradise.image" (dict "Values" .Values "repository" "auth" "tag" "latest") }}
```

### Values Hierarchy

```
values.yaml (defaults)
  â†“
values-staging.yaml (overrides)
  â†“
--set flags (command-line overrides)
```

**Example:**
```bash
# Use staging values, but override replicas
helm install electronic-paradise . \
  -f values-staging.yaml \
  --set services.auth.replicas=5
```
