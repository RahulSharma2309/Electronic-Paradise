## üéØ Our Mission & Principles

### Mission Statement
**"Enable customers to discover, compare, and purchase certified organic products confidently with a checkout experience that is fast, trustworthy, and transparent."**

**In simple terms:** We want customers to feel confident buying organic products from FreshHarvest Market. They should be able to find what they want, verify the organic certifications, understand what they're buying, and trust that fresh, authentic products will be delivered.

### Our Principles (What We Always Follow)

1. **Clarity over cleverness**
   - We make things simple and understandable, not fancy and confusing

2. **Trust is a feature**
   - Customers must trust us - this is as important as any feature

3. **Fast paths for common journeys**
   - The most common things (browse ‚Üí buy) should be quick and easy

4. **Failure must be safe**
   - If something goes wrong, we handle it gracefully - no lost orders or money

5. **Incremental delivery**
   - We ship improvements regularly, not all at once

## üóÇÔ∏è How Work Is Organized

### Epics (Big Feature Sets)
Think of an **Epic** like a major project. For example:
- **Epic 1:** Enhanced Product Catalog (organic certifications, categories)
- **Epic 2:** Advanced Order Management
- **Epic 3:** Multiple Payment Methods

Each epic takes several sprints to complete.

### PBIs (Product Backlog Items)
Think of a **PBI** like a specific feature or task. For example:
- **PBI 1.1:** Product Categories (Fresh Fruits, Vegetables, Grains, etc.)
- **PBI 1.2:** Product Variants (pack sizes: 1kg, 5kg, 10kg)
- **PBI 1.3:** Dynamic Pricing (seasonal discounts, bulk pricing)

Each PBI is a specific piece of work that can be completed in a sprint.

### Sprints (2-Week Work Periods)
A **Sprint** is a 2-week period where we:
1. Plan what to work on
2. Build it
3. Test it
4. Review it
5. Start the next sprint

**Current sprint:** We're about to start Epic 1, Sprint 1

---

## üöÄ What Happens in Each Sprint?

### Sprint Planning (Day 1)
- Review what we're building this sprint
- Understand the PBIs (work items)
- Ask questions about acceptance criteria
- Commit to what we can complete

### Daily Work (Days 2-9)
- Developers build features
- You review and provide feedback
- Test what's being built
- Answer questions about requirements

### Sprint Review (Day 10)
- Show what was built
- Get feedback
- Celebrate progress
- Plan improvements

### Sprint Retrospective (Day 10)
- What went well?
- What could be better?
- How do we improve?

**Then we start the next sprint!**

## üí° Common Questions

### "What if I don't understand something technical?"
**Answer:** That's okay! Ask questions. The team is here to help. You don't need to understand every technical detail - you need to understand what we're building and why.

### "How do I know if a feature is 'done'?"
**Answer:** Check the acceptance criteria in the PBI document. If all criteria are met, it's done. See [Definition of Done](../3-product-owner/DEFINITION_OF_DONE.md) for details.

### "What if we need to change the plan?"
**Answer:** That's normal! Plans change. We adjust the roadmap as we learn. The important thing is to communicate changes clearly.

### "How do I prioritize features?"
**Answer:** Think about:
- What do users need most?
- What helps us achieve our goals?
- What's the biggest risk if we don't build it?
- What can we build quickly that provides value?

### "What if a sprint doesn't go as planned?"
**Answer:** That happens! We learn from it in the retrospective and adjust. Some PBIs might move to the next sprint - that's okay.
